# Security Review 0.10.3 (2026-07-16)

Full review of the auth, sync, and upload paths against `docs/SECURITY.md`
and the OWASP Top 10, per ROADMAP 0.10.3. Reviewed code: `server/src/auth/*`,
`server/src/authz/*`, `server/src/campaigns/*`, `server/src/invites/*`,
`server/src/entities/*`, `server/src/api/contentPacks.ts`,
`server/src/api/conversion/*`, `server/src/http/*`, `server/src/app.ts`,
`server/src/config.ts`, all migrations, and the shared zod schemas those
paths validate with.

## Findings fixed in this review

Each fix landed with a regression test.

1. **Private content-pack read via campaign attachment (medium).**
   `PATCH /api/campaigns/:id` accepted any UUIDs in `packIds` with no
   ownership check, and `GET /api/content-packs/:id` grants read access to
   members of any campaign the pack is attached to. A Keeper who learned a
   stranger's private pack id could attach it to their own campaign and read
   it. Attach now requires the pack to be readable by the Keeper (their own
   upload or the shared library); ids already attached are grandfathered so
   removals keep working.
2. **No `trust proxy` support (medium, deployment).** SECURITY.md section 8
   required it but nothing configured it, so behind a reverse proxy every
   client shared the proxy's IP: the 10/min login bucket throttled the whole
   instance together (trivial lockout DoS) and the fail2ban log lines were
   useless. Added `MOWC_TRUST_PROXY` (unset = off, integer = trusted hops,
   `true` = 1 hop, never "trust everything").
3. **CSRF origin check ignored the port (low/medium).** The check compared
   hostnames only, so a page served from another port of the same host (a
   common self-hosted layout) passed as same-origin. Now compares
   `Origin` host against the request `Host` including the port.
4. **Sync last-write-wins timestamp poisoning (low/medium).** `op.ts` is
   client-supplied and was stored unchecked as the LWW key; a far-future
   timestamp (year 9999) would win every later merge on that entity forever.
   The server now clamps any `ts` more than 5 minutes ahead of its own clock
   (docs/SYNC.md updated). Also added: an op whose entity exists with a
   different `type` is dropped, matching the existing campaign-boundary rule.
5. **Cross-owner pack id collision returned 500 (low).** The duplicate-id
   check on `POST /api/content-packs` was scoped to the uploader, but `id`
   is the table's primary key, so a second user uploading the same id hit an
   unhandled constraint error (500, and a pack-id existence oracle by
   status-code contrast). Now a clean 409 for any existing id.
6. **25 MB conversion body buffered before authorization (low).** The raw
   PDF parser was mounted ahead of `requireAuth`, so unauthenticated and
   non-admin clients could make the server buffer 25 MB per request before
   being refused. The parser now sits behind `requireAuth` and a new
   `requireAdmin` middleware; the router's internal admin check remains as
   defense in depth.
7. **Character strings had no length caps (low, contract).** SECURITY.md
   section 1 requires explicit `.max()` on every string; `Character.name`,
   `look`, `notes`, and the shared `DefIdSchema`/`UserIdSchema` were
   unbounded (only the 1 MB body cap contained them). Caps added
   (name/ids 100, look/notes 5000).
8. **Expired sessions were never deleted (low, hygiene).** `touchSession`
   rejected expired rows but left them in the table forever, despite the
   migration's "expiry sweep on read" comment. Expired rows are now deleted
   when seen.
9. **`npm audit` was soft-fail in CI (Phase 10 gate).** Upgraded vitest 2 to
   4 and applied `npm audit fix`, clearing the one critical and one high
   advisory (both in the dev-only vite/vitest chain; production dependencies
   were already clean). CI now hard-fails at `--audit-level=high`.

## Verified sound (no change needed)

- **Passwords/sessions**: Argon2id via the argon2 package; session tokens
  256-bit random, stored SHA-256-hashed, rolling 30-day expiry, real
  revocation on logout. No signing secret exists or is needed
  (SECURITY.md section 6 corrected; it referenced a `MOWC_SESSION_SECRET`
  that was never part of the design).
- **Authorization**: all campaign, invite, sync, and entity access routes
  through `createAuthz`/`requireKeeper`; non-members get 404 (no UUID
  probing oracle); sync push checks `canEdit` against both the current owner
  and the resulting owner, so a hunter cannot reassign a character; sync
  pull filters per row; cross-campaign entity ids are rejected.
- **Injection**: every statement is a prepared statement with bound
  parameters; no string-built SQL. All API inputs zod-validated strict;
  `__proto__`/`constructor`/`prototype` keys rejected at the parse boundary
  on both JSON-assigning paths (packs, sync). Svelte escaping plus the
  `{@html}` ban (enforced by `npm run check`) covers pack-content XSS.
- **Conversion endpoint**: fixed argv subprocess (no shell), stdin/stdout
  pipes only (no temp files), 30 s timeout, 4 MB output cap, magic-byte
  check, single-flight, 10/hour per user, stateless.
- **Rate limiting**: global 300/min/IP, auth 10/min/IP, invite redemption
  10/min/IP, sync push 60/min/user with a 500-op batch cap, conversion
  10/hour/user. 429s and failed logins are logged with IP for fail2ban.
- **Headers**: frame-ancestors via header, hash-based script-src via
  SvelteKit's per-page meta CSP, nosniff, no-referrer, HSTS under TLS.
- **Invites**: 128-bit random, hashed at rest, 72 h expiry, revocable,
  redemption rate-limited.

## Accepted risks and deferred items

- **Email enumeration**: register returns 409 for an existing email, and
  login timing differs slightly for unknown emails. Accepted for a
  self-hosted app; a signup-disclosure-free flow needs email delivery,
  which the app deliberately does not have.
- **Open registration**: anyone who can reach the instance can register.
  The deployment guidance (LAN or reverse proxy with its own access
  control) is the boundary; an invite-gated registration mode would be a
  new feature, not a review fix.
- **`applied_ops` pruning** (30-day retention is documented but no pruning
  job exists) and **sync backoff/status UI**: Phase 7 scope per the
  SECURITY.md gate table.
- **Pull pagination**: `GET /api/sync/:campaignId` returns every row since
  the cursor with no page cap; bounded today by campaign size. Phase 10.4
  (performance pass) scope.
- **`revealed` is not yet wired into the sync pull filter context**: the
  only synced type (`character`) has no `revealed` field, so this currently
  fails closed. Phase 5 MUST pass `revealed` through `accessCtx` in
  `server/src/entities/router.ts` when it adds Keeper entities.
- **Session cookie `Expires` is not refreshed** on rolling touches, so an
  active browser session still ends 30 days after login. Functional nit,
  not a security issue.
- **Remaining `npm audit` lows** (4): SvelteKit's bundled `cookie` <0.7.0
  advisory, fix arrives with SvelteKit; dev-and-build-time only.
- **fail2ban filter/jail files** (`deploy/fail2ban/`) are still to be
  shipped; the log lines they would consume already exist. Optional per the
  production checklist.
